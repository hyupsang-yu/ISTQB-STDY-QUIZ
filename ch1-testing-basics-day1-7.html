DAY 1
테스팅이란 무엇인가?
📌 테스팅의 정의
소프트웨어 테스팅은 소프트웨어 산출물과 관련 작업 산출물을 평가하고, 이들이 지정된 요구사항을 충족하는지 결정하며, 목적에 부합하는지 입증하고, 결함을 발견하는 일련의 활동입니다.
⚠️ 중요한 오해 바로잡기
❌ 흔한 오해
테스팅 = 소프트웨어 실행 (X)
테스팅은 단순히 소프트웨어를 실행하는 것이 아닙니다!
테스팅은 계획, 모니터링, 분석, 설계, 구현, 실행, 완료 등 다양한 활동을 포함합니다.
🎯 테스팅의 7가지 목적
1
작업 산출물 품질 평가 — 요구사항, 사용자 스토리, 설계, 코드 등을 평가
2
요구사항 충족 여부 베리피케이션 — 명세대로 만들어졌는지 확인
3
완성 및 기대 충족 여부 밸리데이션 — 사용자 실제 요구를 만족하는지 확인
4
결함 유발 및 장애 발견 — 동적 테스팅으로 장애를 유발하여 결함 발견
5
품질 수준에 대한 자신감 제공
6
리스크 수준 완화 — 품질 부족으로 인한 비즈니스 리스크 줄이기
7
법적·계약적 요구사항 준수
🔍 베리피케이션 vs 밸리데이션
구분베리피케이션밸리데이션
질문"제대로 만들었나?""제대로 된 걸 만들었나?"
초점명세/요구사항 충족사용자 실제 필요 충족
예시설계서대로 코딩됐는지 확인사용자가 실제 쓸 수 있는지 확인
⚡ 동적 테스팅 vs 정적 테스팅
구분동적 테스팅정적 테스팅
소프트웨어 실행✅ 실행함❌ 실행 안 함
방법테스트 케이스 실행리뷰, 정적 분석
대상실행 가능한 코드코드, 문서, 요구사항 등
결함 발견장애를 통해 간접 발견결함 직접 식별
💡 시험 포인트
정적 테스팅은 소프트웨어를 실행하지 않고도 코드, 설계서, 요구사항 명세서 등을 검토하는 것입니다. 코드 리뷰가 대표적인 예입니다.

DAY 2
테스팅과 디버깅
🔧 테스팅 vs 디버깅
테스팅과 디버깅은 완전히 별개의 활동입니다. 이 둘을 혼동하면 안 됩니다.
구분테스팅디버깅
목적결함/장애 식별결함 원인 찾아 제거
담당자테스터개발자
활동테스트 실행 → 장애 발견원인 분석 → 수정
🔄 디버깅 프로세스 순서 (시험 필수!)
① 장애 재현 → ② 분석(결함 찾기) → ③ 결함 수정
💡 시험 포인트
디버깅 순서는 장애 재현 → 분석 → 수정입니다. 순서가 바뀌면 오답!
✅ 확인 테스팅 vs 리그레션 테스팅
확인 테스팅 (Confirmation Testing)
  • 결함을 수정한 후, 그 결함이 실제로 해결됐는지 확인하는 테스팅
  • 원래 결함을 발견했던 동일한 테스터가 수행하는 것이 바람직함
  • 목적: 결함 수정 확인
리그레션 테스팅 (Regression Testing)
  • 변경(수정, 새 기능 추가 등) 후 다른 부분에 부정적 영향이 없는지 확인
  • 반복적으로 실행되므로 자동화에 적합
  • 목적: 변경으로 인한 부작용 확인
💡 핵심 구분
확인 테스팅 = "이 버그 고쳤어?" 확인
리그레션 테스팅 = "고치다가 다른 데 망가진 거 없어?" 확인
📍 정적 테스팅에서의 디버깅
정적 테스팅(리뷰 등)에서 결함을 발견한 경우, 디버깅은 해당 결함을 제거하는 것에 중점을 둡니다. 장애를 재현할 필요 없이 결함을 직접 찾았기 때문입니다.

DAY 3
테스팅이 왜 필요한가?
💼 테스팅이 성공에 기여하는 방법
  • 테스팅은 개발 초기부터 참여해 요구사항, 설계, 코드의 품질을 높임
  • 결함을 일찍 발견할수록 수정 비용이 저렴함 (조기 테스팅 원리)
  • 릴리스 전 소프트웨어 품질에 대한 자신감 제공
  • 계약적·법적 요구사항이나 규제 표준 준수 증명
🏢 테스팅 vs 품질보증(QA)
구분테스팅품질보증(QA)
초점제품 지향적프로세스 지향적
접근법교정(결함 발견 후 수정)예방(좋은 프로세스로 결함 방지)
목적특정 산출물의 품질 측정개발/테스트 프로세스 개선
결과 활용결함 수정에 활용프로세스 개선 피드백으로 활용
💡 핵심
QA는 "좋은 프로세스를 따르면 좋은 제품이 나온다"는 가정을 기반으로 합니다. 테스팅의 결과는 QA 피드백으로도 활용됩니다.
⚡ 오류 → 결함 → 장애 → 근본원인
1
오류(Error) — 사람(개발자, 분석가 등)의 실수. 결함을 만들어냄
2
결함(Defect/Bug) — 오류로 인해 코드나 산출물에 생긴 문제. 실행 시 장애를 일으킬 수 있음
3
장애(Failure) — 결함이 있는 코드가 실행됐을 때 나타나는 증상. 단, 결함이 있어도 장애가 안 날 수도 있음
4
근본원인(Root Cause) — 오류가 발생한 근본적인 이유. 근본원인 분석으로 프로세스 개선 가능
⚠️ 주의
결함이 있다고 반드시 장애가 발생하지는 않습니다. 해당 코드가 실행되지 않거나 특정 조건이 안 맞으면 장애가 안 날 수 있습니다.
장애의 원인 — 사람만이 아니다!
  • 개발자의 코딩 실수 (가장 일반적)
  • 방사선, 전자기장 등 환경 조건으로 인한 펌웨어 문제도 장애 원인이 될 수 있음

DAY 4
테스팅의 7가지 원리
📜 7가지 원리 전체 (반드시 암기!)
1
테스팅은 결함의 존재를 밝히는 것이지 결함 없음을 증명하지 않는다
테스팅으로 결함이 없음을 보장할 수 없습니다. 잔존 결함이 있을 수 있습니다.
2
완벽한 테스팅은 불가능하다
모든 입력 조합, 경로를 다 테스트하는 건 불가능. 리스크와 우선순위 기반으로 접근해야 합니다.
3
조기 테스팅으로 시간과 비용을 절약한다 (Shift Left)
결함은 일찍 발견할수록 수정 비용이 적습니다. 정적 테스팅도 조기 테스팅의 일환입니다.
4
결함은 집중된다 (Defect Clustering)
결함의 대부분은 소수의 컴포넌트에 집중됩니다. 파레토 원리(80:20)와 유사합니다.
5
테스트 효과는 감소한다 (살충제 패러독스)
같은 테스트를 반복하면 새로운 결함을 발견하는 능력이 감소합니다. 주기적으로 테스트를 갱신해야 합니다.
6
테스팅은 정황에 의존한다
모든 상황에 맞는 단 하나의 테스팅 접근법은 없습니다. 안전 중시 소프트웨어 vs 전자상거래는 다르게 접근해야 합니다.
7
결함 부재는 궤변이다
결함을 다 수정해도 사용자 기대/요구를 못 맞추면 실패한 소프트웨어입니다. 밸리데이션이 중요한 이유입니다.
🐛 살충제 패러독스 심화
같은 테스트를 계속 실행하면 마치 살충제를 계속 뿌리면 벌레가 내성이 생기듯이, 새로운 결함을 발견하는 효과가 점점 줄어듭니다.
💡 극복 방법
기존 테스트 케이스와 데이터를 정기적으로 수정·보완하고, 새로운 테스트 케이스를 작성해야 합니다.

DAY 5
테스트 활동과 업무
🔄 테스트 프로세스 7단계
💡 중요
이 활동들은 순차적이 아닌 반복적·병렬로 수행됩니다!
1
테스트 계획 (Test Planning)
테스트 목적 정의, 접근법 선택, 자원/일정/비용 계획
2
테스트 모니터링·제어 (Monitoring & Control)
모니터링: 진행상황 정보 수집 / 제어: 목적 달성을 위한 조치
3
테스트 분석 (Test Analysis) — "무엇을?"
테스트 베이시스를 분석해 테스트 컨디션 도출
4
테스트 설계 (Test Design) — "어떻게?"
테스트 컨디션을 테스트 케이스로 구체화, 기법 선정
5
테스트 구현 (Test Implementation)
테스트 스크립트, 테스트 데이터, 테스트 환경 준비
6
테스트 실행 (Test Execution)
테스트 케이스 실행, 결과 기록, 이상 사항 보고
7
테스트 완료 (Test Completion)
완료 보고서 작성, 테스트웨어 보관, 교훈 도출
📋 분석 vs 설계 vs 구현 핵심 구분
활동핵심 질문산출물
테스트 분석무엇을 테스트?테스트 컨디션
테스트 설계어떻게 테스트?테스트 케이스
테스트 구현준비됐나?테스트 스크립트, 데이터

DAY 6
테스트웨어와 추적성
📦 테스트웨어란?
테스트 활동의 결과로 생성되는 모든 산출물을 테스트웨어라 합니다.
활동주요 산출물
테스트 계획테스트 계획서, 리스크 관리 대장
테스트 분석우선순위가 지정된 테스트 컨디션
테스트 설계테스트 케이스, 테스트 차터
테스트 구현테스트 절차, 테스트 스위트, 테스트 데이터
테스트 실행테스트 로그, 결함 보고서
테스트 완료테스트 완료 보고서, 교훈 문서
🔗 추적성(Traceability)이란?
테스트 베이시스 요소 ↔ 테스트웨어 ↔ 테스트 결과 ↔ 결함 간의 연관 관계를 유지하는 것입니다.
추적성의 이점
  • 커버리지 평가 — 요구사항이 얼마나 테스트됐는지 확인 가능
  • 변경 영향 파악 — 요구사항 변경 시 영향받는 테스트 케이스 식별
  • IT 감사 지원 — 감사 기준 충족 증명
  • 이해관계자 소통 — 진행 상황과 품질 수준을 명확히 전달
💡 핵심
추적성이 없으면 "이 요구사항은 테스트됐나요?"라는 질문에 답하기 어렵습니다.

DAY 7
테스팅 역할과 필수 기술
👥 두 가지 주요 역할
구분테스트 관리자테스터
주요 활동계획, 모니터링, 제어, 완료분석, 설계, 구현, 실행
초점관리·전략적기술·실무적
🤝 전체 팀 접근법 (Whole Team Approach)
  • 모든 팀원이 품질에 대한 책임을 공유
  • 팀원들은 작업 공간을 공유하고 협업
  • 테스터와 개발자, 비즈니스 담당자가 함께 작업
⚠️ 전체 팀 접근법이 적합하지 않은 경우
안전이 치명적인 시스템(항공, 의료 등)에서는 높은 수준의 독립적 테스팅이 필요할 수 있습니다.
🎯 테스팅 독립성 수준 (낮음 → 높음)
1
저자(개발자) 자신이 테스트 — 독립성 가장 낮음
2
같은 팀의 동료가 테스트
3
다른 팀/조직 내 별도 테스트팀이 테스트
4
외부 조직(아웃소싱)이 테스트 — 독립성 가장 높음
💡 독립성이 높으면?
인지 편향 없이 결함을 더 많이 발견할 수 있습니다. 하지만 개발팀과 소통이 어려워질 수 있다는 단점도 있습니다.
🛠️ 테스터 필수 기술
분석적 사고
복잡한 시스템을 분석하고 문제를 구조화
꼼꼼함
세부사항에 주의를 기울이는 능력
의사소통
나쁜 소식도 명확히 전달하는 능력
기술 지식
테스트 도구 사용 및 자동화 능력
💡 의사소통이 중요한 이유
결함을 보고할 때 개발자가 방어적이 될 수 있습니다. 테스터는 나쁜 소식을 전달하는 역할이므로 의사소통 기술이 매우 중요합니다.
