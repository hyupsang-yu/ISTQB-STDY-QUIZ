DAY 18
동등분할 / 경계값 분석
🔷 동등 분할 (Equivalence Partitioning)
테스트 대상을 같은 방식으로 처리되는 그룹(분할)으로 나누고, 각 분할에서 대표값 하나만 테스트하는 기법입니다.
핵심 원칙
  • 같은 분할 안의 모든 값은 동일하게 처리된다고 가정
  • 각 분할당 최소 1개의 값만 테스트해도 충분
  • 분할들은 서로 겹치지 않아야 함
유효 분할 vs 비유효 분할
구분유효 분할비유효 분할
의미처리해야 하는 정상 값거부/무시해야 하는 비정상 값
예 (1~100)1, 50, 100 등0, -1, 101 등
테스트 필요?✅ 반드시✅ 반드시
📝 예시: 나이 입력 (1~120세)
유효 분할: 1~120 → 대표값: 50
비유효 분할 1: 0 이하 → 대표값: -1
비유효 분할 2: 121 이상 → 대표값: 150
💡 이치 초이스 커버리지 (Each Choice Coverage)
모든 분할 집합의 각 분할을 최소 한 번은 실행하는 것. 100% EP 커버리지의 기준입니다.
📏 경계값 분석 (Boundary Value Analysis)
동등 분할의 경계(boundary)에서 결함이 자주 발생한다는 원칙 기반의 기법입니다.
2값 BVA vs 3값 BVA
구분2값 BVA3값 BVA
테스트 값경계값 + 바로 인접값경계값 + 양쪽 이웃값
엄격성보통더 엄격
발견 결함off-by-one 오류잘못 배치된 경계 발견
📝 예시: 1~10 범위 (2값 BVA)
하한 경계: 1(경계), 0(바로 아래)
상한 경계: 10(경계), 11(바로 위)
→ 테스트값: 0, 1, 10, 11
📝 예시: 1~10 범위 (3값 BVA)
하한: 0(아래), 1(경계), 2(위)
상한: 9(아래), 10(경계), 11(위)
→ 테스트값: 0, 1, 2, 9, 10, 11
💡 경계값 분석이 잡는 오류
'if (x <= 10)' 을 'if (x == 10)'으로 잘못 구현한 경우, 3값 BVA의 x=9 테스트로 발견 가능!

DAY 19
결정 테이블 / 상태 전이
📋 결정 테이블 테스팅
여러 조건의 조합에 따라 다른 동작이 발생하는 시스템을 테스트하는 기법입니다.
결정 테이블 구성
  • 조건(Condition): 입력 조건들 (T/F)
  • 동작(Action): 조건 조합에 따른 결과
  • 규칙(Rule): 하나의 조건 조합 컬럼
특수 기호 의미
기호의미
T조건이 참(True)
F조건이 거짓(False)
–해당 조건이 결과에 영향 없음 (무관)
N/A해당 규칙에서 조건이 실행 불가
💡 100% 커버리지
모든 실현 가능한 조건 조합(열)을 최소 한 번씩 실행해야 합니다.
🔄 상태 전이 테스팅
시스템이 상태(State)를 가지고, 이벤트에 따라 상태가 전이되는 시스템을 테스트하는 기법입니다.
상태 전이 표기법
이벤트 [가드 조건] / 동작

예: 로그인 [비밀번호 맞음] / 세션 생성
3가지 커버리지 기준 (약한 → 강한 순)
1
모든 상태 커버리지 — 모든 상태를 최소 한 번 방문
가장 약한 기준
2
유효 전이 커버리지 (0-스위치) — 모든 유효한 전이를 최소 한 번 실행
가장 많이 사용되는 기준 ⭐
3
모든 전이 커버리지 — 유효 + 비유효 전이까지 모두
가장 강한 기준, 비유효 전이 = 잘못된 입력 테스트
💡 상태 테이블 vs 상태 다이어그램
상태 테이블은 비유효 전이를 빈 셀로 명확히 보여줌. 다이어그램은 유효 전이만 표시.

DAY 20
화이트박스 기법
🔬 화이트박스 테스팅이란?
코드의 내부 구조를 기반으로 테스트를 설계하는 기법입니다. 구조 기반 테스팅이라고도 합니다.
📊 구문 커버리지 (Statement Coverage)
공식
구문 커버리지 = (실행한 구문 수 / 전체 실행 가능 구문 수) × 100%
  • 모든 실행 가능한 구문을 최소 한 번 실행
  • 100% 달성해도 모든 결정 논리가 테스트됨을 보장하지 않음
🌿 분기 커버리지 (Branch Coverage)
공식
분기 커버리지 = (실행한 분기 수 / 전체 분기 수) × 100%
  • 모든 조건 분기(True/False)를 최소 한 번 실행
  • if문의 참인 경우와 거짓인 경우 모두 테스트
💡 매우 중요! 관계 정리
100% 분기 커버리지 달성 → 100% 구문 커버리지 자동 달성 ✅
100% 구문 커버리지 달성 → 100% 분기 커버리지 보장 안 됨 ❌

분기 커버리지 > 구문 커버리지 (분기가 더 강한 기준)
⚠️ 화이트박스 테스팅의 한계
❌ 화이트박스가 찾지 못하는 것
구현되지 않은 요구사항의 누락 결함을 식별하지 못합니다. 코드에 없는 기능은 테스트할 수 없기 때문입니다.
💡 화이트박스의 가치
객관적인 커버리지 측정값 제공, 아직 실행되지 않은 코드 영역 식별 가능

DAY 21
경험 기반 기법
🎯 오류 추정 (Error Guessing)
테스터의 경험과 직관을 활용하여 발생 가능한 결함을 예측하고 테스트를 설계하는 기법입니다.
결함 공격 (Fault Attack)
발생 가능한 오류, 결함, 장애 목록을 만들고 이를 기반으로 테스트를 설계하는 체계적 접근법입니다.
경험 기반 지식의 출처
  • 애플리케이션의 과거 동작 이력
  • 개발자가 범하기 쉬운 오류 유형
  • 다른 유사 애플리케이션에서 발생한 장애 유형
🔍 탐색적 테스팅 (Exploratory Testing)
테스트 설계, 실행, 평가를 동시에 수행하는 기법입니다.
테스트 차터 (Test Charter)
탐색적 테스팅의 목적이 정의된 지침 문서입니다. 무엇을 탐색할지, 어떻게 할지, 무엇을 찾을지를 정의합니다.
세션 기반 테스팅
탐색적 테스팅을 구조화하는 방법으로, 정해진 시간(세션) 안에 차터를 기반으로 테스팅을 수행합니다.
탐색적 테스팅이 유용한 상황
  • 명세가 부족하거나 부적합한 경우
  • 테스팅에 시간적 압박이 있는 경우
  • 다른 테스트 기법을 보완하는 용도
✅ 체크리스트 기반 테스팅
경험으로 작성된 체크리스트를 기반으로 테스트를 수행하는 기법입니다.
  • 체크리스트가 상위 수준이면 실제 테스트는 조금씩 달라질 수 있음
  • 일부 항목은 효과가 떨어지면 정기적으로 갱신 필요

📖 사용자 스토리 3C
C
Card (카드) — 사용자 스토리를 설명하는 매체 (포스트잇 등)
C
Conversation (대화) — 소프트웨어를 어떻게 사용할지에 대한 설명
C
Confirmation (확인) — 인수 조건. 스토리가 완료됐는지 확인하는 기준
📝 사용자 스토리 형식
As a [역할]
I want to [목표]
So that [이유/가치]
🎯 INVEST 기준 (좋은 사용자 스토리)
I
Independent — 다른 스토리와 독립적
N
Negotiable — 협상 가능 (고정된 계약이 아님)

E
Estimable — 추정 가능
S
Small — 작고 구현 가능한 크기
T
Testable — 테스트 가능
📋 인수 조건 작성 형식
시나리오 기반 (Given/When/Then)
Given 사용자가 장바구니 페이지에 있을 때
When 상품의 수량을 0으로 변경하면
Then 해당 상품이 장바구니에서 제거된다
규칙 기반
목록 또는 입력-출력 매핑 형태로 작성합니다.
