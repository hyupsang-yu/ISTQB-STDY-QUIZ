DAY 8
SDLC와 테스팅 관계
📐 순차적 개발 모델
폭포수 모델 (Waterfall)
  • 각 단계가 순서대로 진행, 이전 단계로 돌아가기 어려움
  • 테스팅은 개발 완료 후에 수행 → 늦은 결함 발견 → 높은 수정 비용
V-모델 (검증 및 확인 모델)
  • 각 개발 단계에 대응하는 테스트 레벨이 존재
  • 테스터는 초기 단계부터 요구사항 리뷰, 테스트 분석·설계에 참여
  • 동적 테스팅은 여전히 나중에 수행되지만, 정적 테스팅은 초기부터 수행
💡 V-모델 대응 관계
요구사항 ↔ 인수 테스팅
시스템 설계 ↔ 시스템 테스팅
아키텍처 설계 ↔ 통합 테스팅
상세 설계 ↔ 컴포넌트 테스팅
🔄 반복적·점진적 개발 모델
  • 스크럼, 칸반, 애자일 등이 대표적
  • 각 반복 주기(스프린트)마다 모든 테스트 레벨 수행 가능
  • 가벼운 산출물, 테스트 자동화 선호
  • 리그레션 테스팅이 매우 중요해짐
⚙️ SDLC가 테스팅에 미치는 영향
  • 테스트 활동의 범위와 시기
  • 테스트 문서의 상세화 수준
  • 테스트 기법과 접근법 선택
  • 테스트 자동화 범위
  • 테스터의 역할과 책임
💡 핵심 원칙
어떤 SDLC를 사용하든, 모든 개발 활동에 상응하는 테스트 활동이 존재해야 합니다.

DAY 9
TDD / ATDD / BDD
🔴 TDD (테스트 주도 개발)
① 테스트 먼저 작성 → ② 실패 확인 → ③ 코드 작성 → ④ 테스트 통과 → ⑤ 리팩토링
  • 코드 작성 전에 테스트를 먼저 작성
  • 리팩토링: 기능 변경 없이 코드 구조를 개선하는 것
  • 주로 컴포넌트(단위) 테스팅 수준에서 사용
🟢 ATDD (인수 테스트 주도 개발)
① 명세 워크숍 → ② 인수 조건 정의 → ③ 테스트 작성 → ④ 구현
  • 인수 조건에서 테스트를 도출
  • 시스템 설계 프로세스 중에 테스트를 작성
  • 비즈니스 관점에서 테스트를 정의
🔵 BDD (행위 주도 개발)
Given / When / Then 형식
Given 사용자가 로그인된 상태에서
When 장바구니에 상품을 추가하면
Then 장바구니 수량이 1 증가해야 한다
  • 이해관계자가 이해할 수 있는 자연어로 작성
  • 개발자, 테스터, 비즈니스 담당자 간 공통 이해 촉진
📊 TDD / ATDD / BDD 공통점
  • 모두 코드 작성 전에 테스트를 정의 → 조기 테스팅 원리 적용
  • 모두 시프트 레프트를 따름
  • 작성된 테스트는 자동화 테스트로 유지·활용 가능

DAY 10
DevOps와 시프트 레프트
⬅️ 시프트 레프트 (Shift Left)
테스트를 SDLC의 더 초기에 수행하는 접근법입니다. 왼쪽으로 이동한다는 뜻으로, 타임라인상 앞당긴다는 의미입니다.
시프트 레프트 프랙티스
  • 테스터 관점에서 명세를 리뷰한다
  • 코드 작성 전에 테스트 케이스를 작성 (TDD)
  • CI/CD에서 정적 분석 도구를 사용
  • 가능한 한 컴포넌트 테스팅에서 비기능 테스팅을 수행
⚠️ 주의
시프트 레프트는 후반부 테스트를 무시해도 된다는 뜻이 아닙니다. 더 일찍 시작하는 것일 뿐, 후반 테스트도 여전히 필요합니다.
🚀 DevOps와 테스팅
DevOps의 테스팅 이점
  • 코드 품질에 대한 빠른 피드백 제공
  • 자동 리그레션 테스트의 규모와 범위 증가
  • 안정적인 테스트 환경 구축 촉진
  • CI/CD를 통한 지속적 통합·배포
DevOps의 리스크/어려움
  • 배포 파이프라인 정의 및 설정 필요
  • CI/CD 도구 도입 및 유지보수 비용
  • 테스트 자동화를 위한 추가 자원 필요
⚠️ 흔한 오해
DevOps가 수동 테스팅을 완전히 제거하지는 않습니다.
🔁 회고 (Retrospective)
반복 주기 종료 시 팀이 모여 수행하는 프로세스 개선 활동입니다.
  • 무엇이 성공적이었는가? (유지할 것)
  • 무엇이 부족했는가? (개선할 것)
  • 향후 어떻게 개선할 것인가?
💡 테스팅 관점의 이점
테스트 효과성/효율성 향상, 테스트웨어 품질 향상, 팀 학습 촉진

DAY 11
테스트 레벨
🏗️ 5가지 테스트 레벨
1
컴포넌트 테스팅 (단위 테스팅)
• 개별 컴포넌트를 독립적으로 테스트
• 일반적으로 개발자가 수행
• 목적: 컴포넌트 기능·비기능 특성 검증
2
컴포넌트 통합 테스팅
• 컴포넌트 간의 인터페이스와 상호작용 테스트
• 목적: 인터페이스 결함 발견
3
시스템 테스팅
• 전체 시스템의 기능·비기능 동작 검증
• E2E(엔드투엔드) 테스팅 포함
• 독립적인 테스트팀이 수행하는 것이 좋음
4
시스템 통합 테스팅
• 다른 시스템·외부 서비스와의 인터페이스 테스트
• API, 웹 서비스 등 외부 연동 검증
5
인수 테스팅
• 사용자 비즈니스 요구 충족 여부 확인 (밸리데이션)
• 유형: UAT, 운영 인수, 계약 인수, 알파/베타 테스팅
🎯 테스트 레벨 구분 속성
테스트 대상
무엇을 테스트하나
테스트 목적
왜 테스트하나
테스트 베이시스
무엇을 근거로
결함과 장애
어떤 결함을 찾나

DAY 12
테스트 유형과 확인/리그레션
📋 4가지 테스트 유형
1
기능 테스팅 — "무엇을" 하는가?
시스템이 기능적 요구사항을 만족하는지 확인. 블랙박스 기법 사용.
2
비기능 테스팅 — "얼마나 잘" 하는가?
성능, 보안, 사용성, 신뢰성 등 ISO 25010 품질 특성 검증
3
블랙박스 테스팅 — 명세 기반
내부 구조를 모르고 입출력만으로 테스트. 동등분할, 경계값 분석 등 사용.
4
화이트박스 테스팅 — 구조 기반
내부 구조(코드, 아키텍처)를 기반으로 테스트. 구문/분기 커버리지 등 사용.
💡 ISO 25010 비기능 품질 특성 예시
수행 효율성, 호환성, 사용성, 신뢰도, 보안, 유지보수성, 이식성
🔁 확인 vs 리그레션 (복습)
확인 테스팅리그레션 테스팅
목적결함 수정 확인부정적 영향 없는지 확인
언제결함 수정 직후변경이 있을 때마다
자동화상황에 따라자동화에 매우 적합

DAY 13
유지보수 테스팅
🔧 유지보수 테스팅이란?
이미 운영 중인 시스템에 변경이 발생했을 때 수행하는 테스팅입니다.
유지보수 테스팅의 3가지 계기
1
수정사항 — 계획된 기능 개선, 릴리스, 긴급 핫픽스 등
2
업그레이드/마이그레이션 — 운영 환경 변경, 다른 플랫폼으로 이전
→ 새 환경 관련 테스트 + 데이터 변환 테스트 필요
3
단종(Retirement) — 시스템 폐기 시
→ 데이터 보관(archiving) 테스트, 복원/복구 절차 테스트 필요
📏 유지보수 테스팅 범위 결정 요소
  • 변경의 리스크 수준 — 높을수록 더 많은 테스팅
  • 기존 시스템의 크기 — 클수록 영향 범위가 넓음
  • 변경사항의 크기 — 변경이 클수록 더 많은 테스팅
💡 영향도 분석
유지보수 변경 전에 영향도 분석을 수행하여 시스템 다른 부분에 미칠 영향을 파악합니다. 이를 통해 리그레션 테스팅 범위를 결정합니다.

DAY 14
1·2장 복습 및 정리
📝 1장 핵심 정리
꼭 기억해야 할 것들
  • 테스팅 = 실행만이 아닌 계획~완료까지 전체 활동
  • 7가지 원리 모두 암기 (특히 살충제 패러독스, 결함 집중, 결함 부재의 궤변)
  • 오류 → 결함 → 장애 순서와 차이
  • 테스팅(제품 지향) vs QA(프로세스 지향)
  • 확인 테스팅 vs 리그레션 테스팅
  • 독립성: 저자 자신 → 동료 → 외부팀 → 외부조직
📝 2장 핵심 정리
꼭 기억해야 할 것들
  • V-모델: 각 개발 단계에 대응하는 테스트 레벨 존재
  • TDD: 테스트 먼저, ATDD: 인수 조건에서, BDD: Given/When/Then
  • 시프트 레프트: 후반 테스트를 없애는 게 아니라 더 일찍 시작
  • 5가지 테스트 레벨: 컴포넌트→컴포넌트통합→시스템→시스템통합→인수
  • 유지보수 테스팅 계기: 수정/업그레이드/단종
🎯 자주 틀리는 문제 포인트
⚠️ 함정 주의
1. 시프트 레프트 = 후반 테스트 무시 → 틀림!
2. DevOps = 수동 테스팅 완전 제거 → 틀림!
3. 결함 있으면 반드시 장애 발생 → 틀림!
4. 테스팅 = 소프트웨어 실행만 → 틀림!
