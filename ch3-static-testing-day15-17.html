DAY 15
정적 테스팅 기초
📖 정적 테스팅이란?
소프트웨어를 실행하지 않고 작업 산출물을 평가하는 테스팅입니다. 리뷰와 정적 분석이 포함됩니다.
검사 가능한 작업 산출물
  • 요구사항 명세서, 사용자 스토리
  • 아키텍처 설계서, 상세 설계서
  • 소스 코드
  • 테스트 계획서, 테스트 케이스
  • 프로젝트 계획서, 예산 및 일정
⚠️ 정적 테스팅이 적합하지 않은 경우
사람이 해석하기 어려운 산출물(예: 바이너리 파일)은 정적 테스팅이 어렵습니다.
⚡ 정적 vs 동적 테스팅
구분정적 테스팅동적 테스팅
실행 여부❌ 실행 안 함✅ 실행함
결함 발견결함 직접 식별장애를 통해 발견
적용 대상실행 불가 산출물 포함실행 가능 코드만
특수 결함도달 불가 코드 발견 가능런타임 오류 발견
비용일반적으로 저렴상대적으로 비쌈
💡 정적 테스팅만 찾을 수 있는 결함
• 도달 불가능한 코드 (Dead Code)
• 요구사항의 불일치·모호성·모순
• 잘못된 인터페이스 명세
• 보안 취약점 패턴
🔧 정적 분석
도구를 사용하여 소프트웨어를 실행하지 않고 자동으로 코드를 분석하는 방법입니다.
  • 테스트 케이스 없이 수행 가능
  • CI/CD 파이프라인에 통합하여 자동화 가능
  • 코딩 표준 위반, 보안 취약점, 코드 복잡성 등 발견

DAY 16
리뷰 프로세스
🔄 리뷰 프로세스 5단계 (순서 암기!)
1
계획 (Planning)
리뷰 범위, 목적, 일정, 참여자 선정, 진입/완료 조건 정의
2
리뷰 착수 (Review Initiation)
모든 참여자와 사항이 준비됐는지 확인. 리뷰 시작 전 체크리스트 확인
3
개별 리뷰 (Individual Review)
각 리뷰어가 독립적으로 이상 사항을 찾음
4
논의 및 분석 (Issue Communication)
이상 사항의 상태, 담당자, 필요 조치를 논의하고 결정
5
수정 및 보고 (Fixing and Reporting)
결함 수정, 완료 조건 충족 여부 확인, 결과 보고
💡 시험 포인트
순서: 계획 → 착수 → 개별 리뷰 → 논의/분석 → 수정/보고
특히 "개별 리뷰"가 "논의" 앞에 온다는 것을 기억!
👥 리뷰에서의 역할
관
관리자 — 리뷰 내용 결정, 필요 자원 제공
저
저자 — 리뷰 대상 산출물 작성자, 결함 수정
중
중재자(퍼실리테이터) — 리뷰 회의 진행, 시간 관리, 안전한 분위기 조성
서
서기(기록자) — 이상 사항 수집 및 기록, 결정 사항 문서화
리
리뷰어 — 이상 사항 식별, 결함 발견
리더
리뷰 리더 — 리뷰 전체 책임, 참여자 선정, 리뷰 일정 관리
🌟 조기 피드백의 이점
  • 요구사항에 대한 오해 방지
  • 요구사항 변경을 조기에 이해하고 구현 가능
  • 초기 단계 결함 발견 → 낮은 수정 비용

DAY 17
리뷰 유형과 성공 요소
📊 4가지 리뷰 유형 (공식성 낮은→높은 순)
① 비공식 리뷰 가장 비공식
• 정의된 프로세스 없음, 공식 문서 불필요
• 이메일, 메신저로 간단히 검토 요청
• 목적: 간단한 결함 발견, 동료 검토
② 워크쓰루 (Walkthrough)
• 저자가 리더가 되어 참여자를 안내
• 개별 리뷰 수행 선택적
• 목적: 정보 공유, 결함 발견, 교육, 토론
③ 기술 리뷰 (Technical Review)
• 기술 전문가가 리뷰어로 참여
• 중재자(퍼실리테이터)가 리더
• 목적: 기술적 합의 도출, 결함 발견
④ 인스펙션 (Inspection) 가장 공식적
• 가장 공식적이고 체계적인 리뷰
• 메트릭 수집하여 프로세스 개선에 활용
• 저자는 리뷰 리더나 서기가 될 수 없음 ← 시험 필수!
• 목적: 최대한 많은 결함 발견
💡 공식성 순서
비공식 리뷰 < 워크쓰루 < 기술 리뷰 < 인스펙션
✅ 리뷰 성공 요소
  • 명확한 목표와 측정 가능한 완료 조건 정의
  • 적절한 리뷰 유형 선택 (목적에 맞는 유형)
  • 작은 단위로 나눠서 리뷰 (집중력 유지)
  • 리뷰어에게 충분한 준비 시간 제공
  • 관리자의 지원
  • 사람이 아닌 산출물에 집중하는 문화
⚠️ 리뷰의 목적은?
리뷰는 참가자를 평가하거나 비난하는 것이 아닙니다. 산출물의 품질을 높이는 것이 목적입니다.
