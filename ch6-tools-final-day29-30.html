✅ 테스트 자동화의 6가지 효과
  • 반복적 수작업 줄여 시간 절약
  • 수동 테스팅보다 일관성 있고 재현 가능
  • 결과에 대한 객관적 평가 제공
  • 도구가 수집한 정보에 더 쉽게 접근
  • 수동 테스팅보다 실행 시간 단축
  • 테스터가 더 창의적인 테스트 설계에 집중 가능
⚠️ 테스트 자동화의 7가지 리스크
  • 도구 효과에 대한 비현실적인 기대
  • 도구 도입에 드는 시간, 비용, 노력 과소평가
  • 테스트 자동화를 달성하기 어려울 정도로 노력이 과소평가
  • 도구에 지나치게 의존하여 비판적 사고 무시
  • 수동이 더 적합한 곳에 자동화 적용
  • 오픈소스 소프트웨어의 지원 중단 가능성
  • 호환성 문제 등 기술적 어려움
⚠️ 핵심 오해
도구를 도입한다고 자동으로 성공이 보장되지 않습니다. 도입, 유지보수, 교육에 추가 노력이 필요합니다.

🎯 자주 출제되는 비교 문제
테스팅 vs 디버깅
테스팅 = 결함 식별 / 디버깅 = 결함 원인 제거
확인 테스팅 vs 리그레션 테스팅
확인 = 수정 확인 / 리그레션 = 부작용 확인
베리피케이션 vs 밸리데이션
베리피케이션 = "제대로 만들었나?" / 밸리데이션 = "제대로 된 걸 만들었나?"
분기 커버리지 vs 구문 커버리지
분기 100% → 구문 100% 보장 O / 구문 100% → 분기 100% 보장 X
2값 BVA vs 3값 BVA
3값이 더 엄격 / 3값 = 경계값 + 양쪽 이웃값
🚨 자주 틀리는 함정 문제
❌ 이것들은 모두 틀린 내용!
• 테스팅 = 소프트웨어 실행만 → 틀림
• 테스팅으로 결함 없음 증명 가능 → 틀림
• 결함 있으면 반드시 장애 발생 → 틀림
• 시프트 레프트 = 후반 테스트 불필요 → 틀림
• DevOps = 수동 테스팅 완전 제거 → 틀림
• 도구 도입 = 자동으로 성공 보장 → 틀림
• 인스펙션에서 저자가 리더/서기 가능 → 틀림
• 구문 100% = 분기 100% 보장 → 틀림
